# Deploying Nextcloud AIO on GCP Compute Engine

<Callout tone="info" title="TL;DR">
  Use Cloudflare DNS-only (grey cloud) for initial setup and certificates, and never point `NEXTCLOUD_DATADIR` at a shared disk root; use a dedicated subdirectory.
</Callout>

This is a concise, reproducible record of deploying Nextcloud All-in-One (AIO) on a Google Cloud Compute Engine VM (Debian 12) with Cloudflare DNS (`nextcloud.microalchemy.xyz`), plus the issues encountered and fixes applied.

## Architecture

- GCE VM (Debian 12)
- Docker Engine on the VM
- Nextcloud AIO mastercontainer manages the stack (Nextcloud, Apache, DB, Redis, etc.)
- Separate persistent disk mounted for Nextcloud data
- Cloudflare DNS (DNS-only recommended during initial setup)

## Setup instructions

### 1) GCP firewall rules and instance tags

Add a network tag to the VM (example: `nextcloud-aio`). Create an ingress firewall rule targeting that tag:

- TCP: `80`, `443`, `8080`, `8443`
- TCP/UDP: `3478` (Talk; only if Talk is enabled)
- TCP: `22` (SSH; ideally restricted to your IP)

### 2) Attach and mount a persistent disk for data

Attach a larger persistent disk (example: `/dev/sdb`), then format and mount it:

```bash
sudo mkfs.ext4 -F -L ncdata /dev/sdb
sudo mkdir -p /mnt/ncdata
sudo mount /dev/sdb /mnt/ncdata
sudo blkid /dev/sdb
```

Persist it using the UUID:

```bash
sudo sh -c 'echo "UUID=<UUID> /mnt/ncdata ext4 defaults,nofail 0 2" >> /etc/fstab'
sudo mount -a
```

### 3) Install Docker

```bash
sudo apt-get update
curl -fsSL https://get.docker.com | sudo sh
sudo systemctl enable --now docker
```

### 4) Create a dedicated Nextcloud data directory (critical)

Do not point AIO's `NEXTCLOUD_DATADIR` at the disk root. Use a dedicated subdirectory:

<Callout tone="warning" title="Avoid data directory corruption">
  If `NEXTCLOUD_DATADIR` overlaps with Docker/containerd storage, AIO can recursively `chown` paths you never intended, breaking the host and other containers.
</Callout>

```bash
sudo mkdir -p /mnt/ncdata/nextcloud-data
sudo chown 33:0 /mnt/ncdata/nextcloud-data
sudo chmod 750 /mnt/ncdata/nextcloud-data
```

### 5) Start the AIO mastercontainer

```bash
docker volume create nextcloud_aio_mastercontainer

docker run -d \
  --init \
  --sig-proxy=false \
  --name nextcloud-aio-mastercontainer \
  --restart always \
  -p 80:80 -p 8080:8080 -p 8443:8443 \
  -v nextcloud_aio_mastercontainer:/mnt/docker-aio-config \
  -v /var/run/docker.sock:/var/run/docker.sock:ro \
  -e NEXTCLOUD_DATADIR="/mnt/ncdata/nextcloud-data" \
  ghcr.io/nextcloud-releases/all-in-one:latest
```

### 6) Cloudflare DNS

Create the DNS record:

- `A nextcloud.microalchemy.xyz -> <VM external IP>`

During initial setup and certificate issuance, use DNS-only (grey cloud). Proxied DNS (orange cloud) can break AIO's domain validation/on-demand TLS workflows.

### 7) Complete setup in the browser

- AIO UI: `https://<VM_IP>:8080`
- Certificate issuance/setup endpoint: `https://nextcloud.microalchemy.xyz:8443`
- Final Nextcloud URL: `https://nextcloud.microalchemy.xyz/` (port 443)

## Verification

On the VM:

```bash
sudo ss -lntp | grep -E '(:22 |:80 |:443 |:8080 |:8443 )'
docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"
```

From your machine:

```bash
curl -I http://nextcloud.microalchemy.xyz/
curl -Ik https://nextcloud.microalchemy.xyz/
```

Expected results:

- HTTP redirects to HTTPS
- HTTPS serves or redirects to `/login`

## Postmortem

### 1) Cloudflare 525 / TLS handshake failures on `:8443`

Symptom:

- `https://nextcloud.microalchemy.xyz:8443/` returned Cloudflare 525 ("SSL handshake failed").

Cause:

- DNS was proxied (orange cloud), so the hostname resolved to Cloudflare IPs, not the VM.
- AIO's on-demand TLS/domain validation flow rejected or failed under proxying.

Fix:

- Switched the record to DNS-only during setup/certificate issuance.
- Verified `nextcloud.microalchemy.xyz` resolved directly to the VM IP before retrying.

### 2) AIO containers failing to start due to missing images

Symptom:

- AIO could not create containers (e.g., Collabora) because required images were missing.

Cause:

- The required `ghcr.io/nextcloud-releases/aio-*` images were not present locally when AIO attempted creation.

Fix:

- Pulled the required images and re-triggered the AIO "Start containers" flow.

### 3) Permissions corruption: `sudoers owned by uid 33` and Postgres permission errors

Symptom:

- Nextcloud logs: `sudo: /etc/sudoers is owned by uid 33, should be 0`
- Database logs: permission denied errors on Postgres data directories.

Cause:

- Nextcloud startup scripts recursively `chown` `NEXTCLOUD_DATADIR` when a write test fails.
- `NEXTCLOUD_DATADIR` had been set to `/mnt/ncdata` (disk root), and Docker/containerd storage also lived under `/mnt/ncdata`.
- The recursive `chown` changed ownership of Docker/containerd storage and volumes, cascading into failures across containers.

Fix:

- Isolated Nextcloud data under `/mnt/ncdata/nextcloud-data`.
- Kept Docker/containerd storage out of Nextcloud-controlled paths.
- Reset/recreated the AIO stack after corruption.

### 4) External timeouts while ports appeared open

Symptom:

- TCP connections could open, but SSH/HTTP would hang (no banner/response), presenting as timeouts.

Fix:

- Reset the GCE instance and verified listeners, container health, and HTTP responses after reboot.

## Key takeaways

- Never set `NEXTCLOUD_DATADIR` to a shared disk root; always use a dedicated subdirectory.
- Keep Docker/containerd storage separate from application-controlled directories.
- Use Cloudflare DNS-only during initial AIO setup and certificate issuance; proxying can be re-tested later.
- Validate at the application layer (e.g., `curl -I`), not just "port open".
